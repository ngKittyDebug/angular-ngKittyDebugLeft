# Стайлгайд по коммитам

В нашем проекте используется [Conventional Commits](https://www.conventionalcommits.org/) с конфигурацией `@commitlint/config-angular` и расширенным лимитом строки заголовка до **200 символов**.

## Формат сообщения коммита

Каждое сообщение коммита должно иметь следующую структуру:

```text
тип(область): краткое_описание
```

- **тип** — что именно мы сделали (например: `feat`, `fix`, `refactor`).
- **область** — номер/ссылка на задачу из трекера (например, `ISSUE-123` или `link_to_issue`).
- **краткое_описание** — суть изменений (до 200 символов вместе с типом).

> **Примечание:** Мы не пишем подробное описание (body) в самих коммитах. Вся подробная информация, скриншоты и объяснения архитектурных решений должны находиться в описании Pull Request'а.

### Пример правильного коммита:

```text
feat(ISSUE-42): добавлена страница профиля пользователя
```

## Типы коммитов (`type`)

| Тип        | Описание                                                                                          |
| ---------- | ------------------------------------------------------------------------------------------------- |
| `feat`     | Добавление нового функционала.                                                                    |
| `fix`      | Исправление бага.                                                                                 |
| `docs`     | Изменение только документации (README, гайды в папке docs).                                       |
| `style`    | Изменения, не влияющие на смысл кода (пробелы, форматирование, точки с запятой и т.д.).           |
| `refactor` | Изменение кода, которое не исправляет баг и не добавляет новую функцию.                           |
| `perf`     | Изменение кода, улучшающее производительность.                                                    |
| `test`     | Добавление отсутствующих тестов или исправление существующих.                                     |
| `build`    | Изменения, влияющие на систему сборки или внешние зависимости (npm, pnpm, webpack, angular.json). |
| `ci`       | Изменения в конфигурационных файлах и скриптах CI (GitHub Actions).                               |
| `chore`    | Прочие изменения (обновление тулинга, не меняющие src или тесты).                                 |
| `revert`   | Откат предыдущего коммита.                                                                        |

## Правила и рекомендации

1. **`scope` обязателен или крайне желателен**, если коммит привязан к задаче. Пишите туда номер задачи (напр. `feat(AUTH-12)`) или ссылку.
2. **`subject` (тема коммита)**:
   - должна быть краткой и понятной (не более 200 символов в сумме с типом и скоупом).
   - пишется в повелительном наклонении или просто описывает факт (напр. "добавлена кнопка", "исправлен расчет", "add user form").
   - без точки на конце.
3. **Хуки (Husky)**:
   - При каждом коммите проект будет **автоматически полностью отформатирован** (Prettier) и **проверен/исправлен линтером** (ESLint, Stylelint). Изменения от линтера автоматически сохранятся в этот коммит (`git add -u`).
   - Затем пройдет проверка типов (`pnpm typecheck`) и прогон тестов (`pnpm test`). Если что-то упадет — коммит будет отменен, чтобы в историю не попал нерабочий код.
